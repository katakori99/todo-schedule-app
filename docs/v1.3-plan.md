# v1.3 Supabase導入計画・手順書

最終更新: 2026-06-16

## 目的

v1.3 では、v1.2 のローカル保存版を Supabase 保存版へ移行します。

ただし、現在の `localStorage` には重要なデータは入っていないため、v1.2 データの移行機能は作りません。
v1.3 では「ログイン後に Supabase に新規保存する」形へ切り替えます。

## 前提

- Googleログインは使わない
- メール Magic Link ログインを使う
- テストに使うメールアドレスは `nameki.seito@gmail.com`
- v1.2 の localStorage データ移行は不要
- v1.3 完了時のバージョンは `1.3.0`
- v2.0 までは Vercel 公開しない

## v1.3 の完成形

v1.3 完了時点で、アプリは以下の状態になります。

- 未ログイン時はメールログイン画面を表示する
- `nameki.seito@gmail.com` に Magic Link を送れる
- Magic Link クリック後、アプリにログインできる
- ログイン中のユーザーの ToDo だけが表示される
- ログイン中のユーザーの時間割だけが表示される
- ToDo の追加、完了、削除、並び替えが Supabase に保存される
- 時間割の追加、削除が Supabase に保存される
- 時間割の重複は引き続き拒否される
- ログアウトできる
- 読み込み中、未ログイン、保存失敗などの状態が表示される
- `npm run build` が通る

## 重要な設計判断

### localStorage 移行はしない

v1.2 の localStorage データは重要ではないため、以下は実装しません。

- localStorage から Supabase への移行ボタン
- migration marker
- localStorage データの重複取り込み
- 移行件数表示

ただし、v1.3 実装中も localStorage を明示的に削除する必要はありません。
単に Supabase 保存へ切り替え、旧データを読まなくなる方針です。

### 時間割DBは `start_hour` / `duration_hours`

最初のロードマップでは `start_time` / `end_time` 案でしたが、v1.2 の実装は以下です。

```ts
type ScheduleItem = {
  id: string;
  date: string;
  startHour: number;
  duration: 1 | 2 | 3;
  titleHtml: string;
  createdAt: number;
  updatedAt: number;
};
```

そのため Supabase 側も `start_hour` / `duration_hours` で設計します。
この方が現在のUI、1h/2h/3h切り替え、重複判定と一致します。

## 1-4. SQLを実行する

Supabase dashboard で以下を開きます。

```text

```

以下のSQLをそのまま実行してください。

```sql
create extension if not exists pgcrypto;
create extension if not exists btree_gist;

create or replace function public.set_updated_at()
returns trigger
language plpgsql
as $$
begin
  new.updated_at = now();
  return new;
end;
$$;

create table if not exists public.todo_items (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  content_html text not null default '',
  done boolean not null default false,
  order_index integer not null default 0,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists public.schedule_items (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  date date not null,
  title_html text not null default '',
  start_hour smallint not null check (start_hour >= 6 and start_hour < 24),
  duration_hours smallint not null check (duration_hours in (1, 2, 3)),
  order_index integer not null default 0,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint schedule_items_ends_within_day check (start_hour + duration_hours <= 24),
  constraint schedule_items_no_overlap
    exclude using gist (
      user_id with =,
      date with =,
      int4range(start_hour, start_hour + duration_hours, '[)') with &&
    )
);

create index if not exists todo_items_user_order_idx
  on public.todo_items (user_id, order_index, created_at);

create index if not exists schedule_items_user_date_start_idx
  on public.schedule_items (user_id, date, start_hour);

grant usage on schema public to authenticated;
grant select, insert, update, delete on public.todo_items to authenticated;
grant select, insert, update, delete on public.schedule_items to authenticated;

drop trigger if exists todo_items_set_updated_at on public.todo_items;
create trigger todo_items_set_updated_at
before update on public.todo_items
for each row
execute function public.set_updated_at();

drop trigger if exists schedule_items_set_updated_at on public.schedule_items;
create trigger schedule_items_set_updated_at
before update on public.schedule_items
for each row
execute function public.set_updated_at();

alter table public.todo_items enable row level security;
alter table public.schedule_items enable row level security;

drop policy if exists "todo_items_select_own" on public.todo_items;
create policy "todo_items_select_own"
on public.todo_items
for select
to authenticated
using ((select auth.uid()) is not null and (select auth.uid()) = user_id);

drop policy if exists "todo_items_insert_own" on public.todo_items;
create policy "todo_items_insert_own"
on public.todo_items
for insert
to authenticated
with check ((select auth.uid()) is not null and (select auth.uid()) = user_id);

drop policy if exists "todo_items_update_own" on public.todo_items;
create policy "todo_items_update_own"
on public.todo_items
for update
to authenticated
using ((select auth.uid()) is not null and (select auth.uid()) = user_id)
with check ((select auth.uid()) is not null and (select auth.uid()) = user_id);

drop policy if exists "todo_items_delete_own" on public.todo_items;
create policy "todo_items_delete_own"
on public.todo_items
for delete
to authenticated
using ((select auth.uid()) is not null and (select auth.uid()) = user_id);

drop policy if exists "schedule_items_select_own" on public.schedule_items;
create policy "schedule_items_select_own"
on public.schedule_items
for select
to authenticated
using ((select auth.uid()) is not null and (select auth.uid()) = user_id);

drop policy if exists "schedule_items_insert_own" on public.schedule_items;
create policy "schedule_items_insert_own"
on public.schedule_items
for insert
to authenticated
with check ((select auth.uid()) is not null and (select auth.uid()) = user_id);

drop policy if exists "schedule_items_update_own" on public.schedule_items;
create policy "schedule_items_update_own"
on public.schedule_items
for update
to authenticated
using ((select auth.uid()) is not null and (select auth.uid()) = user_id)
with check ((select auth.uid()) is not null and (select auth.uid()) = user_id);

drop policy if exists "schedule_items_delete_own" on public.schedule_items;
create policy "schedule_items_delete_own"
on public.schedule_items
for delete
to authenticated
using ((select auth.uid()) is not null and (select auth.uid()) = user_id);
```

このSQLで作るもの:

- `todo_items` テーブル
- `schedule_items` テーブル
- 更新日時を自動更新する trigger
- `authenticated` ロールへのテーブル操作権限
- ユーザーごとにデータを分離する RLS policy
- 時間割の重複をDB側でも防ぐ制約

もしログイン後に以下のエラーが出た場合:

```text
permission denied for table todo_items
```

SQL Editor で以下だけを追加実行してください。

```sql
grant usage on schema public to authenticated;
grant select, insert, update, delete on public.todo_items to authenticated;
grant select, insert, update, delete on public.schedule_items to authenticated;
```

これは RLS を無効化する設定ではありません。
`authenticated` ロールにテーブル操作の入口を開き、実際に自分のデータだけ操作できるかは上の RLS policy で制限します。

## 1-5. Supabase の Project URL と Publishable key を取得する

ここでは `.env.local` に入れる2つの値を取得します。

必要な値:

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
```

この2つは、Supabase の画面上で別々の場所にあることがあります。

## 1-5-a. Project URL を取得する

Supabase dashboard で対象 project を開きます。

左メニューから以下を開きます。

```text
Integrations -> Data API
```

そこで `API URL` または `Project URL` と表示されている値をコピーします。

形式は以下のどちらかです。

```text
https://xxxxxxxxxxxxxxxxxxxx.supabase.co
```

または:

```text
https://xxxxxxxxxxxxxxxxxxxx.supabase.co/rest/v1
```

確認ポイント:

- `https://` で始まる
- `.supabase.co` を含む
- `.env.local` に入れるときは `/rest/v1` を外す
- `apikey=` などのクエリ文字列も付けない

例:

```text
画面に表示されている値:
https://abcdefghijklmnopqrst.supabase.co/rest/v1

.env.local に入れる値:
https://abcdefghijklmnopqrst.supabase.co
```

もし `Integrations -> Data API` が見つからない場合:

1. Supabase dashboard の project 一覧に戻る
2. 対象 project を開き直す
3. 左メニューの検索欄があれば `Data API` と入力する
4. 画面上部またはサイドバー付近に `Connect` ボタンがあれば開く
5. Connect 画面内の `Project URL` / `SUPABASE_URL` / `VITE_SUPABASE_URL` を探す
6. それでも見つからない場合、project ref からURLを作れます

`Connect` 画面で以下のような環境変数例が出ている場合:

```text
SUPABASE_URL=https://abcdefghijklmnopqrst.supabase.co
SUPABASE_PUBLISHABLE_KEY=sb_publishable_xxxxx
```

または:

```text
VITE_SUPABASE_URL=https://abcdefghijklmnopqrst.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=sb_publishable_xxxxx
```

その `https://...supabase.co` の部分を `VITE_SUPABASE_URL` に使います。

project ref が分かる場合、URL は以下の形です。

```text
https://PROJECT_REF.supabase.co
```

例:

```text
https://abcdefghijklmnopqrst.supabase.co
```

この値を `.env.local` の `VITE_SUPABASE_URL` に入れます。

## 1-5-b. Publishable key を取得する

Supabase dashboard で対象 project を開きます。

左メニューから以下を開きます。

```text
Settings -> API Keys
```

あなたが今見ている `API Keys` 画面がここです。

表示されているタブまたはセクションで、以下を探します。

```text
Publishable and secret API keys
```

その中の以下を使います。

```text
Publishable key
```

あなたの画面では、次のように表示されています。

```text
New publishable key
Name: default
API Key: sb_publishable_...
```

この `sb_publishable_...` で始まる値をコピーします。

この値を `.env.local` の `VITE_SUPABASE_PUBLISHABLE_KEY` に入れます。

重要:

- `Publishable key` はブラウザ用です
- `Secret keys` は使いません
- `service_role` も使いません
- `Secret key` / `service_role` は RLS を回避できる強い鍵なので、ブラウザアプリに入れてはいけません

公式 docs でも、API URL は `Integrations -> Data API`、API Keys は `Settings -> API Keys` にあると説明されています。

公式 docs では、新しいキー方式でブラウザなどの client-side operation には `Publishable key` を使い、server-side operation には `Secret key` を使う、と説明されています。

## 1-6-b. `.env.local` に書く内容

`.env.local` の中身は2行だけです。

```text
VITE_SUPABASE_URL=https://YOUR_PROJECT_REF.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=YOUR_PUBLISHABLE_KEY
```

`YOUR_PROJECT_REF` と `YOUR_PUBLISHABLE_KEY` はそのまま書かず、実際の値に置き換えてください。


あとは `VITE_SUPABASE_URL=` の右側に Project URL を入れてください。

## 1-6-c. よくある間違い

以下は間違いです。

```text
VITE_SUPABASE_URL=abcdefghijklmnopqrst
```

理由: `https://` と `.supabase.co` がないため。

以下も間違いです。

```text
VITE_SUPABASE_URL=https://abcdefghijklmnopqrst.supabase.co/rest/v1
```

理由: Supabase client に渡す URL は project URL だけでよく、`/rest/v1` は不要です。

以下も間違いです。

```text
VITE_SUPABASE_PUBLISHABLE_KEY=sb_secret_xxxxxxxxx
```

理由: `sb_secret_...` は Secret key です。ブラウザ用ではありません。

## 2-4. ToDo を Supabase 保存にする

実装内容:

- `todo_items` から読み込む
- ToDo 追加時に insert
- 完了切替時に update
- 1件削除時に delete

読み込みクエリ案:

```ts
supabase
  .from("todo_items")
  .select("*")
  .eq("user_id", user.id)
  .order("order_index", { ascending: true })
  .order("created_at", { ascending: true });
```

```ts
supabase
  .from("schedule_items")
```

## 2-7. localStorage の扱いを整理する

v1.3 では localStorage データ移行をしません。

対応方針:

- ToDo と時間割は Supabase から読む

- `cafe-todo.tasks.v1` と `cafe-todo.schedule.v1` は読まない
- `cafe-todo.view.v1` は画面状態保存として残してよい
- 旧 localStorage データを削除するUIは作らない

## 4-4. RLS 確認

最低限、以下を確認します。

- ログイン中ユーザーのデータだけを取得している
- query に `.eq("user_id", user.id)` が入っている
- Supabase 側で RLS が有効になっている前提で動作する

余裕があれば、別メールアドレスで User B を作って、User A とデータが分離されることを確認します。

## ロールバック方針

v1.3 実装で大きな問題が出た場合:

1. コードを v1.2 の状態へ戻す
2. Supabase project は残してよい
3. localStorage データは重要ではないため、移行や復元は考えない
4. SQL は実データがなければ再実行してよい

実データが入った後は、テーブル削除や project 削除は慎重に判断してください。
