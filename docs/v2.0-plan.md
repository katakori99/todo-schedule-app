# v2.0 Vercel公開 + Supabase安定版 手順書

最終更新: 2026-06-17

## 目的

v2.0 では、現在ローカルで動いている Supabase 保存版を Vercel に公開し、PC とスマホから同じデータを使える安定版にします。

v2.0 の中心は新機能追加ではなく、以下です。

- Vercel 本番URLでアプリを開ける
- 本番URLで Magic Link ログインできる
- ToDo と時間割が Supabase に保存される
- PC とスマホで同じアカウントのデータが同期される
- 本番URL、環境変数、Supabase Auth Redirect URL の設定漏れをなくす

## 現在の前提

- v1.4.0 まで完了
- Supabase project 作成済み
- Email Magic Link ログイン動作済み
- ToDo / 時間割は Supabase 保存
- RLS policy と `authenticated` grant は設定済み
- localStorage の重要データ移行は不要
- `.env.local` はローカル用で、git には入れない
- `npm run build` は成功している

## v2.0 完了条件

- Vercel の Production URL でアプリが開ける
- Vercel の Production URL で Magic Link ログインできる
- ToDo の追加、編集、削除が本番で動く
- 時間割の追加、編集、削除が本番で動く
- Markdown と数式表示が本番で動く
- PC とスマホで同じアカウントのデータが同期される
- リロード後もログイン状態とデータが維持される
- Supabase Auth の Site URL / Redirect URLs が本番向けになっている
- `npm run build` が通る
- `package.json` が `2.0.0`

## 重要な判断

### localStorage 移行はしない

現在の方針どおり、localStorage に重要データはないため、v2.0 でも移行機能は作りません。

### ブラウザに入れてよいキーは Publishable key だけ

Vercel に設定するキーは以下です。

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
```

入れてはいけない値:

```text
sb_secret_...
service_role
SUPABASE_SECRET_KEY
SUPABASE_SERVICE_ROLE_KEY
```

理由:

- `sb_publishable_...` はブラウザアプリで使うためのキー
- `sb_secret_...` や `service_role` は RLS を回避できる強いキー
- Secret key はブラウザ、GitHub、チャット、URL、公開ファイルに入れない

### Vite の環境変数は `VITE_` prefix が必要

このアプリは Vite なので、ブラウザ側コードから読む環境変数は `VITE_` で始めます。

正しい:

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
```

間違い:

```text
SUPABASE_URL
SUPABASE_PUBLISHABLE_KEY
```

## 1. Codex の v2.0 前チェック

現在のチェック結果:

- `package.json` version: `1.4.0`
- Vite project
- Build command: `npm run build`
- Build output: `dist`
- Vercel向けに特別な serverless function は不要
- `.env.local` は `.gitignore` 対象
- `.env.example` を追加済み
- `VITE_SUPABASE_URL` / `VITE_SUPABASE_PUBLISHABLE_KEY` を使用中
- `emailRedirectTo` は `window.location.origin` を使っているため、本番では Vercel URL へ戻る

## 2. あなたのタスク: GitHubへrepoを用意する

Vercel は GitHub repo から import するのが一番扱いやすいです。

### 2-1. GitHubで新規repositoryを作る

1. GitHub を開く
2. `New repository` を押す
3. repository name を決める
   例:

```text
todo-schedule-app
```

4. Public / Private はどちらでもよい
5. README は追加しなくてよい
6. `.gitignore` も追加しなくてよい
7. repository を作成する

### 2-2. Codexへrepository URLを伝える

作成後、GitHub repository のURLをCodexへ伝えてください。

例:

```text
https://github.com/nameki/todo-schedule-app
```

ここから先の `git remote add` / commit / push は Codex が行えます。

## 3. あなたのタスク: Vercel projectを作成する

GitHubへpushできた後に行います。

### 3-1. Vercelへログインする

1. https://vercel.com/dashboard を開く
2. GitHub account でログインする

### 3-2. Projectをimportする

1. `Add New...` または `New Project` を押す
2. GitHub repository 一覧からこのアプリの repo を選ぶ
3. `Import` を押す

### 3-3. Framework / Build設定を確認する

Vercel が Vite を認識した場合、概ね以下になります。

```text
Framework Preset: Vite
Build Command: npm run build
Output Directory: dist
Install Command: npm install
Root Directory: ./
```

もし自動認識されていなければ、上の値に合わせてください。

## 4. あなたのタスク: Vercel環境変数を設定する

Vercel の import 画面、または Project 作成後の `Settings -> Environment Variables` で設定します。

追加する変数は2つだけです。

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
```

### 4-1. `VITE_SUPABASE_URL`

値は Supabase project URL です。

形式:

```text
https://YOUR_PROJECT_REF.supabase.co
```

注意:

- `/rest/v1` は付けない
- 最後に余計な空白を入れない

### 4-2. `VITE_SUPABASE_PUBLISHABLE_KEY`

値は `sb_publishable_` で始まる Publishable key です。

形式:

```text
sb_publishable_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

注意:

- `sb_secret_` は絶対に入れない
- `service_role` は絶対に入れない
- Legacy `anon` key でも動く可能性はあるが、今回は Publishable key を使う

### 4-3. Environment の選択

まずは以下すべてに同じ値を入れてよいです。

```text
Production
Preview
Development
```

少なくとも `Production` には必ず入れてください。

重要:

Vercel の環境変数は、変更後の新しいデプロイにだけ反映されます。
設定を直した場合は、必ず redeploy してください。

## 5. あなたのタスク: Production Deployする

ここでは、Vercel 上で最初の本番公開をします。

### 5-1. Deploy を開始する

Vercel の Project 作成画面で、環境変数を2つ入れ終わったら、画面下または右下にある以下のボタンを押します。

```text
Deploy
```

ボタン名が少し違う場合もあります。

例:

```text
Create
Deploy Project
Continue to Deploy
```

意味は同じです。Project 作成を完了して、ビルドを開始するボタンです。

### 5-2. Build が終わるまで待つ

Deploy を押すと、Vercel が自動で以下を実行します。

```text
npm install
npm run build
```

画面には build log が流れます。

成功した場合、以下のような表示になります。

```text
Congratulations!
Ready
Production Deployment
```

または、緑色の成功表示になります。

失敗した場合は、赤いエラー表示になります。
その場合は、エラー画面をそのまま触らずに、Codexへ以下を伝えてください。

```text
Vercel の build が失敗しました。
```

そのうえで、エラー文のスクリーンショットか、エラー文を貼ってください。

### 5-3. Production URL を探す

Deploy が成功すると、Vercel が公開URLを作ります。

URLはだいたい以下の形です。

```text
https://todo-schedule-app.vercel.app
```

探す場所の例:

- Deploy 完了画面の `Visit` ボタン
- Project 画面上部の `Domains`
- Project 画面の `Deployment` 詳細
- `Production` と表示されている deployment のURL

注意:

Vercel には複数のURLが出ることがあります。

例:

```text
https://todo-schedule-app.vercel.app
https://todo-schedule-app-git-main-katakori99.vercel.app
https://todo-schedule-app-abc123-katakori99.vercel.app
```

v2.0でまず使うのは、基本的に短い Production URL です。

優先:

```text
https://todo-schedule-app.vercel.app
```

### 5-4. Production URL を開いて確認する

そのURLをブラウザで開きます。

この時点では、まだ Supabase Auth の本番URL設定が終わっていないため、Magic Link ログインは失敗する可能性があります。
ここでは、まず以下だけ確認できれば十分です。

- アプリのログイン画面が表示される
- 真っ白画面ではない
- `Supabase設定を確認してください` が出ていない

もし `Supabase設定を確認してください` が出たら、Vercel の Environment Variables 設定が間違っています。

確認する値:

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
```

### 5-5. CodexへURLを伝える

Production URL が分かったら、Codexへ次の形で伝えてください。

```text
Vercel Production URL は https://xxxx.vercel.app です。
```

この時点では、まだSupabase側を触らなくてよいです。
CodexがそのURLに合わせて、次に入れるべき Supabase Auth 設定を具体値で返します。

## 6. Codex のタスク: Supabase Auth URL設定値を確定する

あなたから Production URL を受け取ったら、Codex が以下を具体値で返します。

```text
Site URL:
https://xxxx.vercel.app

Redirect URLs:
https://xxxx.vercel.app
https://xxxx.vercel.app/**
http://127.0.0.1:5173/**
http://localhost:5173/**
```

Preview Deployments も Magic Link で確認したい場合は、Vercel account/team slug に合わせて preview wildcard も追加します。

例:

```text
https://*-YOUR_TEAM_OR_ACCOUNT_SLUG.vercel.app/**
```

ただし、本番利用だけなら Production URL の exact URL と `/**` が優先です。

## 7. あなたのタスク: Supabase Auth URLを本番向けに設定する

ここは、Magic Link の戻り先を `localhost` から Vercel 本番URLへ変える作業です。

### 7-1. Supabase Dashboard を開く

1. https://supabase.com/dashboard を開く
2. このアプリで使っている Supabase project を開く
3. 左メニューの `Authentication` を開く
4. `URL Configuration` を開く

表示名が少し違う場合があります。

近い名前:

```text
Authentication -> URL Configuration
Authentication -> Configuration -> URL Configuration
Authentication -> Settings -> URL Configuration
```

### 7-2. Site URL を変更する

`Site URL` という欄を探します。

現在はローカルURLになっている可能性があります。

例:

```text
http://127.0.0.1:5173
```

これを Vercel Production URL に変更します。

例:

```text
https://todo-schedule-app.vercel.app
```

注意:

- 最後に `/` があっても大きな問題にはなりにくいが、手順書ではなしに統一する
- `http` ではなく `https`
- Vercel の Preview URL ではなく、Production URL を入れる

### 7-3. Redirect URLs を追加する

`Redirect URLs` という欄を探します。

ここは1行ずつURLを追加する形式の場合があります。
または、複数行テキスト欄の場合もあります。

Codexから具体値をもらったら、それをそのまま追加してください。

例:

```text
https://todo-schedule-app.vercel.app
https://todo-schedule-app.vercel.app/**
http://127.0.0.1:5173/**
http://localhost:5173/**
```

意味:

- 1行目: 本番URLそのもの
- 2行目: 本番URL以下のパス全部
- 3行目: ローカル開発用
- 4行目: ローカル開発用

すでに `http://127.0.0.1:5173/**` や `http://localhost:5173/**` が入っている場合は、消さずに残してください。

### 7-4. 保存する

画面内の `Save` / `Update` / `Save changes` を押します。

保存後、Supabase側の設定は完了です。

### 7-5. Magic Link を本番URLから送る

Vercel Production URLを開きます。

例:

```text
https://todo-schedule-app.vercel.app
```

ログイン画面でメールアドレスを確認し、Magic Link を送信します。

```text
nameki.seito@gmail.com
```

届いたメールのリンクを開いたあと、戻り先が以下になれば成功です。

```text
https://todo-schedule-app.vercel.app
```

もし `http://127.0.0.1:5173` に戻った場合は、Supabase の Site URL / Redirect URLs がまだ本番向けになっていません。

## 8. Codex のタスク: 本番動作確認

本番URLとSupabase設定が終わったら、Codexが動作確認します。

あなたは、まず以下をCodexへ伝えてください。

```text
Supabase Auth URL設定が終わりました。本番URLでログインできました。
```

または、ログインできない場合は、止まった場所を伝えてください。

例:

```text
Magic Linkメールが届きません。
```

```text
メールは届きましたが、リンクを開くとlocalhostに戻ります。
```

```text
Vercel URLには戻りますが、ログイン画面のままです。
```

Codex側では以下を確認します。

### 8-1. Auth

- 未ログイン時にログイン画面が出る
- Magic Link を送信できる
- メール内リンクから Vercel Production URL に戻る
- ログイン後にアプリ本体が出る
- ログアウトできる
- リロード後もログイン状態が維持される

### 8-2. ToDo

- 追加できる
- Markdown が表示される
- 数式が表示される
- ダブルクリックで編集できる
- 削除できる
- リロード後も残る

### 8-3. 時間割

- 追加できる
- Markdown が表示される
- 数式が表示される
- 長文でもデザインが崩れない
- ダブルクリックで編集できる
- 削除できる
- リロード後も残る

### 8-4. PC / スマホ同期

同じ `nameki.seito@gmail.com` でログインして確認します。

- PCで追加したToDoがスマホで見える
- スマホで追加した時間割がPCで見える
- 片方で削除したデータが、もう片方のリロード後に消える

## 9. v2.0でやらないこと

以下は v2.1 以降で検討します。

- 独自ドメイン
- Custom SMTP
- バックアップ / エクスポートUI
- リアルタイム同期
- 複数ユーザー共有
- パスワードログイン移行
- Googleログイン

ただし、Magic Link の送信制限が運用上つらい場合、Custom SMTP は v2.0中に前倒ししてもよいです。

## 10. トラブルシュート

### Vercelで「Supabase設定を確認してください」と出る

原因:

- Vercel環境変数がない
- 変数名が間違っている
- `VITE_` prefix がない
- 環境変数設定後に redeploy していない

確認:

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
```

### Magic Link後にlocalhostへ戻る

原因:

- Supabase Auth の Site URL がまだローカル
- Redirect URLs に本番URLがない

対応:

```text
Authentication -> URL Configuration
```

で Site URL と Redirect URLs を本番URLへ設定します。

### Magic Link後にログイン画面へ戻る

確認:

- URL に `error_code=otp_expired` がないか
- 古いリンクを開いていないか
- Vercel URL と Supabase Redirect URL が一致しているか

### Buildは成功したが古い環境変数が使われる

Vite は環境変数をビルド時に bundle へ埋め込みます。
Vercel環境変数を変更したら redeploy が必要です。
